# Governance

Ghostshift is being built as an open product, not as a source-available wrapper around a closed roadmap.

## Principles

- Major product direction should be legible in public.
- Core data model changes should be discussed before landing.
- Plugin and storage ecosystems should remain open and documented.
- Maintainers are responsible for protecting contributor trust and repository clarity.

## Decision Process

- Small improvements can be proposed directly through pull requests.
- Cross-cutting technical changes should start as an RFC in `docs/rfcs/`.
- Breaking changes to schemas, plugin APIs, or storage contracts require an RFC.
- Maintainers aim to keep decisions written down, not trapped in chat threads.

## Maintainer Model

Until the project grows, the founding maintainers act as stewards for:

- roadmap direction
- release management
- security response
- schema and API stability

As the contributor base grows, Ghostshift should adopt a lightweight working-group model for:

- spec
- CLI/core
- integrations
- self-hosted operations
